Security & Audits

Security model

Liquid Flow's security is structural, not a setting. The design goal is simple and absolute: Liquid Flow can time and authorize payments, but can never move user funds.

🔒 The invariant. If Liquid Flow's servers and keys were fully compromised, an attacker could not steal or redirect a single user's funds. Everything below exists to make that true.

What Liquid Flow can and cannot do

ActionLiquid Flow
Derive a one-time deposit addressYes
Watch chains & confirm depositsYes
Trigger a scheduled, company-defined payroll payout at its due timeYes
Hold a key that can move a merchant's / campaign's fundsNever
Change a payout's recipient or amount, or release it earlyNever
Withdraw, redirect, or freeze user fundsNever

Stealth addresses (payments & fundraising)

Every payment and donation lands at a fresh one-time stealth address derived from the recipient's published meta-keys. It uses a dual-key scheme (view + spend), the same idea as EIP-5564 / Monero-style stealth:

shared secret  s = H(r · P_view) = H(k_view · R)
one-time key   P = P_spend + s · G          // the deposit address
  • The recipient detects incoming payments with their view key.
  • Only the spend key (returned to the recipient once, never stored by Liquid Flow) can move the funds.
  • Payers and the recipient's main wallet stay unlinked on-chain.

Two curves are implemented so the same model works everywhere:

secp256k1Base, Arc (EVM)0x… keccak address
ed25519Solana, Suibase58 / 0x32-byte
The stealth cryptography is the highest-risk component and is currently unaudited. It must pass a dedicated cryptographic audit before mainnet.

On-chain contracts

Optional smart contracts add on-chain controls. Each is owned by the business; Liquid Flow holds at most a low-privilege timing role.

ContractGuarantee
PaymentGatePer-merchant gate; funds forward to the merchant's own address.
PlatformWallet2-of-N multisig; no single key or person can release funds alone.
SecurePlatformWalletAdds per-token velocity caps, an allowlist delay, and a guardian pause (circuit breaker).
PayrollScheduler / …ERC20Business funds & owns it; the trigger key can only release a due, pre-defined payout to the set recipient. releaseBatch settles a whole cycle in one tx.

Key handling

  • Spend keys (the keys that move funds) are generated for the recipient and shown once. They are never persisted server-side.
  • Operator / trigger key is a low-privilege key used only to submit time-gated payroll releases and confirm deposits. It cannot move funds outside a company-defined schedule. On mainnet it moves to an HSM/KMS.
  • Webhooks are HMAC-SHA256 signed (LF-Signature) so your app can verify authenticity.

Current status & audit plan

⚠️ Testnet, unaudited. Do not use real funds. Mainnet is gated on the following, tracked on the Roadmap:
  • Cryptographic audit — independent review of both stealth schemes (secp256k1 and ed25519), the highest-risk pieces.
  • Smart-contract audit — every contract, every chain, including the ERC-20 payroll and batch release.
  • Formal verification — of the "no unauthorized fund movement" invariant.
  • Testnet bug bounty — before any mainnet deployment.
  • Legal / regulatory review — money-transmission, AML/KYC, travel rule, and the privacy posture, per jurisdiction.

Responsible disclosure

Found something? Please report it privately before public disclosure so we can protect testnet users. A formal security contact and bug-bounty program launch alongside the audit phase.

Liquid Flow is testnet software and not audited. This page describes design intent and current status; it is not a security guarantee.
